01 — The Problem

Why Fraud Detection
Is Hard to Get Right

Standard classification approaches fail on fraud data. Three structural challenges must be solved before a single model is trained.

⚖️

Severe Class Imbalance

Only 3.5% of transactions are fraudulent — a 27:1 ratio. Models trained naively predict "not fraud" for everything and achieve 96.5% accuracy while catching zero fraud cases. Standard accuracy is a lie.

💸

Asymmetric Error Costs

Missing a fraud (False Negative) means absorbing the full loss. Flagging a legitimate transaction (False Positive) creates customer friction and support costs. These errors are not equal — the model must know that.

🎯

Wrong Evaluation Metric

ROC-AUC looks impressive on imbalanced data even for poor models. Precision-Recall AUC is the correct metric — it focuses entirely on the positive (fraud) class and exposes model weaknesses that ROC-AUC hides.

03 — Feature Engineering

Behavioral Signals
Catch What Rules Miss

Raw transaction data is insufficient. The key insight: fraud is defined by deviation from a customer's own baseline, not absolute thresholds. A $2,000 charge is normal for one customer and suspicious for another.

amt_zscore
Z-score of transaction amount relative to this card's own mean and standard deviation. Captures "how unusual is this amount for this specific customer?"
Importance Score: 0.188 · Rank #1
amt_to_card_median_ratio
Transaction amount divided by this card's median spend. A ratio of 12x flags a transaction costing 12× the customer's typical purchase.
Importance Score: 0.162 · Rank #2
card_txn_count_log
Log-transformed transaction count per card. Cards with very few transactions have no established baseline, making them harder to validate and higher risk.
Importance Score: 0.138 · Rank #3
is_night
Binary flag for transactions between midnight and 5am. Fraud rates are elevated by ~2.3× during overnight hours when fraud monitoring is typically lighter.
Importance Score: 0.114 · Rank #4
is_new_card
Flag for cards with only one prior transaction in the dataset. New cards have no behavioral history — deviations cannot be normalized against a personal baseline.
Importance Score: 0.102 · Rank #5
dist1_missing
Missingness indicator for the distance feature. The absence of distance data is itself a signal — fraudulent transactions are more likely to have missing location context.
Importance Score: 0.081 · Rank #6
04 — Methodology

End-to-End Pipeline

Each stage of the pipeline makes deliberate choices to address the specific challenges of fraud data.

01
Data Preparation
Deduplication on transaction ID, per-customer median imputation for missing amounts, stratified train/test split preserving fraud rate
02
Feature Engineering
15+ behavioral features including z-scores, velocity proxies, time signals, and missingness indicators designed to capture anomaly relative to baseline
03
Model Training
XGBoost with scale_pos_weight=27.6 to correct imbalance in the loss function. Hyperparameter tuning via stratified cross-validation with PR-AUC as objective
04
Threshold Optimization
Classification threshold tuned using F-beta (β=2), weighting recall 2× over precision to reflect the higher cost of missed fraud vs false alarms
05 — Implementation

Key Code Patterns

Production-grade implementations of the three core technical decisions in this pipeline.

feature_engineering.py
# Per-customer behavioral deviation features # $500 means different things per cardholder card_stats = df.groupby('card1')['amount'].agg( card_mean='mean', card_median='median', card_std='std', card_count='count' ).reset_index() df = df.merge(card_stats, on='card1', how='left') # Z-score: standard deviations from card mean df['amt_zscore'] = ( (df['amount'] - df['card_mean']) / df['card_std'].replace(0, np.nan) ) # Missing distance is itself a fraud signal df['dist_missing'] = df['dist1'].isna().astype(int) # Per-customer median imputation (not global) df['amount'] = df.groupby('card1')['amount'].transform( lambda x: x.fillna(x.median()) )
model_training.py
# Imbalance ratio drives scale_pos_weight scale_pos_weight = ( (y_train == 0).sum() / (y_train == 1).sum() ) # → 27.6 for 3.5% fraud rate model = xgb.XGBClassifier( n_estimators=300, max_depth=6, learning_rate=0.05, scale_pos_weight=scale_pos_weight, eval_metric='aucpr', # PR-AUC not ROC subsample=0.8, colsample_bytree=0.8 ) model.fit( X_train, y_train, eval_set=[(X_test, y_test)], verbose=False ) # F-beta threshold optimization (β=2) # Weights recall 2× over precision beta = 2 fbetas = [ (1 + beta**2) * p * r / (beta**2 * p + r + 1e-9) for p, r in zip(precisions, recalls) ] threshold = thresholds[np.argmax(fbetas)]
velocity_fraud.sql
-- Velocity fraud: 3+ transactions in 60 min -- Window function approach for production SQL WITH windowed AS ( SELECT customer_id, transaction_id, timestamp, COUNT(*) OVER ( PARTITION BY customer_id ORDER BY timestamp RANGE BETWEEN INTERVAL '60 minutes' PRECEDING AND CURRENT ROW ) AS txn_count_in_window FROM transactions ) SELECT customer_id, transaction_id, timestamp FROM windowed WHERE txn_count_in_window >= 3;
inference.py
# SageMaker inference script # Returns structured fraud assessment def predict_fn(input_data, model): fraud_proba = model.predict_proba( input_data )[:, 1] return fraud_proba def output_fn(prediction, accept): results = [ { 'fraud_score': float(score), 'is_fraud': bool( score >= THRESHOLD ), 'risk_level': ( 'HIGH' if score > 0.70 else 'MEDIUM' if score > 0.35 else 'LOW' ) } for score in prediction ] return json.dumps(results), accept

08 — About This Work

Built to Production
Standards

This project was built to demonstrate end-to-end data science capabilities across the full ML lifecycle — from raw transactional data through feature engineering, model training, evaluation, and cloud deployment — using patterns directly applicable to financial services fraud detection at scale.

The methodology mirrors production work from prior experience building a customer churn prediction model (AUC ~0.97+) at a regional telecom provider, where behavioral feature engineering on transactional data, imbalanced classification, and executive-level deployment were core deliverables.

Fraud detection extends that foundation with higher cost asymmetry, adversarial dynamics, and real-time serving requirements — making it the natural next problem domain.
